# Enforcement Layers: How Skills, Hooks, and Workflows Compose

**Audience:** contributors and operators who want to understand how the framework's four enforcement layers work together and why each layer exists.

**Mode:** Explanation (Diátaxis). This page explains *how* enforcement is achieved at each layer. For *what* each artifact does, see the reference pages. For *why* hooks were chosen over prompts, see [ADR-0002](../adr/0002-hooks-enforce-process-not-prompts.md).

---

## The central problem: instructions vs. enforcement

Stating a rule in a CLAUDE.md produces approximately 25% compliance. The same rule enforced by a hook that can block a response produces approximately 90% compliance. This empirical finding is the foundation of the framework's architecture: every critical rule has a corresponding enforcement artifact, not just a prose statement.

The four layers below are the mechanism. Each layer enforces properties that the layers above it cannot guarantee alone.

## Layer 0: Classifier (UserPromptSubmit + Stop)

**Enforces:** task classification happens before any work begins.

Two hooks cover this layer:
- `user-prompt-submit.py` — injects context on every user message, reinforcing the classification requirement.
- `classifier-field-check.py` — fires at Stop; blocks the response if any required classification field is absent, or if PM is missing from a non-Quick task's MUST DISPATCH list.

The Stop hook is the hard gate. Classification is not verified at the point of promise — it is verified at the point of response delivery. If the session skipped classification, the response is blocked and classification must happen first.

See [task-classification.md](task-classification.md) for why classification is mandatory.

## Layer 1: Process Skills (PreToolUse + PostToolUse + Stop)

**Enforces:** classified tasks follow the correct typed process flow.

Skills are prose procedures — a numbered step sequence the model follows. This layer has below-100% step-compliance, which is why it carries three hooks:
- `skill-routing-check.py` — fires PreToolUse on Skill invocations; blocks if the invoked skill does not match the classifier's TYPE routing.
- `skill-step-reminder.py` — fires PostToolUse after a skill loads; injects step reminders.
- `process-step-check.py` — fires at Stop; blocks if required artifacts (SCOPE block, QA REPORT, PENTEST REPORT) are missing.

**Active architectural direction:** convert process skills into deterministic workflow scripts — routing-as-code — so the dispatch sequence happens by construction rather than by prose compliance. This is a pilot direction, not yet adopted. See [ADR-0004](../adr/0004-routing-as-code-workflow-enforcement.md).

## Layer 2: Agent Delegation (SubagentStart + SubagentStop + Stop)

**Enforces:** declared agents are actually invoked; subagent outputs meet quality standards.

- `subagent-governance.py` — fires SubagentStart; injects behavioral rules into every subagent at spawn.
- `subagent-quality-check.py` — fires SubagentStop; blocks empty, errored, or unstructured agent outputs.
- `dispatch-compliance-check.py` — fires at Stop; verifies every item in MUST DISPATCH was actually invoked via Agent or Skill tool.

The MUST DISPATCH contract is generated by the classifier. The compliance check verifies it was honored. The `DISPATCHES.json` sidecar per skill provides a fallback when the transcript is compacted and the classification block falls outside the hook's read window.

## Layer 3: Tool Safety and Quality (PreToolUse + Stop)

**Enforces:** dangerous commands are blocked; QA reports represent real execution.

- `bash-safety-guard.py` — fires PreToolUse:Bash; blocks `rm -rf`, force-push, credential exposure patterns.
- `work-verification-check.py` — fires at Stop; three checks: (1) hard-blocks QA/pentest reports filed with zero execution tools, (2) hard-blocks premature user escalation with fewer than 3 tool uses, (3) soft-logs zero-work non-Quick tasks.
- `dark-zone-check.py` — fires at Stop; monitors unsupported citations and reasoning failures in areas without tool access.
- `governance-log.py` — fires at Stop; writes all session governance events to a JSONL audit log.

## How the layers compose

The layers are sequential gates, not independent checks. A task must pass Layer 0 (classification) before Layer 1 (process routing) makes sense. Each layer's compliance rate depends on the layer above it working correctly.

When a layer fails, the hook blocks the response and the session must correct at that layer before proceeding. This is deliberate: error accumulation across layers is harder to recover from than a single early block.

All hooks are stateless Python scripts — stdin in, stdout out, no database, no server. The governance log (`governance-log.jsonl`) is the only shared state, append-only.
